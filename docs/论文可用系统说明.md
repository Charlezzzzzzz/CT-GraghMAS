# CT-GraphMAS 系统说明（论文可用稿）

## 1. 系统定位

CT-GraphMAS（Knowledge Graph-Enhanced Multi-Agent System for Computational Thinking-Oriented Programming Tutoring）是面向初中 Python 编程学习场景构建的知识图谱增强多智能体智能辅导系统。系统以计算思维培养为核心目标，将学生问题、课程知识点、典型错误、教学语料和交互记录组织为可检索、可解释、可更新的图谱结构，并通过多智能体协同完成问题诊断、教学推理和反馈呈现。

该系统的设计重点不是替学生直接生成完整代码，而是围绕“定位问题、分析原因、形成支架、保留思考空间、促进迁移反思”的编程学习逻辑，为学生提供分层提示，为教师提供可追溯的教学依据。

## 2. 三层多智能体架构

系统采用“L1 感知与认知整合中枢、L2 计算思维教学推理层、L3 交互呈现层”的三层结构。

L1 感知与认知整合中枢负责接收学生提问、代码片段、错误信息、学习画像和知识图谱相关知识。该层不直接生成面向学生的最终答案，而是形成结构化诊断，包括任务类型、相关知识节点、先修缺口、典型错误、学生认知阶段和智能体调用路径。

L2 计算思维教学推理层由四类智能体构成：P1 问题分解智能体、P2 抽象建模智能体、P3 算法设计智能体和 P4 调试评价智能体。P1 负责引导学生明确输入、处理、输出和边界；P2 负责将真实情境转化为变量、状态和数据结构；P3 负责形成自然语言步骤、伪代码和算法流程；P4 负责围绕错误信息、测试样例和边界条件进行定位与验证。

L3 交互呈现层将 L2 的教学推理结果转译为适合学生理解的反馈。系统支持同伴型、启发导师型和严格教练型三种表达风格，并在输出前执行反代写门控，避免直接提供完整程序、超出学生最近发展区或造成事实误导。

## 3. 核心提示词与工作流设计

系统将实验阶段的三层提示词转化为可配置的系统提示词契约。L1 层提示词要求模型接收学生原始提问或报错、145 个 Python 微观考点的意图映射结果和 2-hop 学情上下文，并以 JSON 形式输出 `student_dilemma`、`cognitive_state`、`target_concept`、`kg_context`、`route` 和 `risk_flags`。这一设计使 L1 只承担诊断与路由职责，避免过早生成答案。

L2 层由 P1、P2、P3、P4 四类提示词构成。P1 从问题规模、模块边界和最小切入点形成“问题分解”支架；P2 从变量、状态和数据容器形成“抽象建模”支架；P3 从执行顺序、条件流转和循环终止形成“算法设计”支架；P4 从假设验证、状态追踪和边界样例形成“评估调试”支架。系统默认根据 L1 路由动态选择智能体组合，实验扩展时也可覆盖 P1-P4 的全部 15 种组合进行对比。

L3 层提示词规定最终回复必须来源于 L2 聚合策略，回复长度控制在 100-500 字，并禁止直接给出完整正确代码。该层将教研策略转化为学生可理解的苏格拉底式追问、局部提示和下一步行动，随后再经过反代写门控检查。

## 4. 知识图谱与语料增强机制

系统内置 CT-GraphMAS 系统节点、三层架构节点、智能体节点、计算思维节点、Python 知识点节点、典型错误节点、教学流程节点和教学门控节点，并加入 145 个带有认知阶段标记的 Python 微观考点节点。节点之间通过 `HAS_LAYER`、`HAS_AGENT`、`SCAFFOLDS`、`PREREQUISITE_OF`、`CAN_CAUSE`、`ORGANIZES`、`KNOWS`、`REQUIRES` 等关系连接，形成可解释的编程学习知识图谱。

教师或研究者可以上传教学文档、课堂材料、案例说明或反思记录。系统会自动完成文本解析、相关知识切分、embedding 和图谱回写。每个文档被切分为若干“相关知识”片段，每个片段生成确定性向量表示，并根据概念命中结果与相应知识节点建立 `MENTIONS` 关系。由此，上传语料不只是被保存为文本，而是转化为能够参与后续检索和对话推理的图谱相关知识。

## 5. 动态相关知识路由策略

系统在对话时采用动态相关知识路由策略。系统首先定位与学生问题直接相关的知识节点、错误节点或语料片段，再扩展到相邻的先修知识、典型错误和支架节点；语料注入只选择少量高相关知识参与反馈生成，避免把完整图谱或大量文档一次性注入对话上下文。

这种设计对应论文实验结论：知识图谱增强并不是信息越多越好。对初中生编程辅导而言，适度的先修链条、典型错误和局部相关知识更有利于保持反馈清晰度，降低认知负荷和系统成本。

## 6. 多智能体路由策略

系统根据学生画像和任务类型动态选择 L2 智能体组合。破冰期新手更强调问题分解和基础调试，因此系统优先调用 P1 与 P4；建构期中等生通常已经具备一定语法基础，但容易卡在流程组织和逻辑关系上，因此优先调用 P1、P3 和 P4；自动化期进阶生更需要抽象建模和方案优化，因此优先调用 P2、P3 和 P4。

当系统识别到任务中存在抽象建模、算法流程或调试错误等特征时，会对路由进行局部调整。这样，多智能体协同不再是简单增加角色数量，而是根据学生认知状态和任务卡点进行有目的的支架调度。

## 7. 反代写与人机协同评价

系统设置反代写门控，重点识别全盘代写、评价场景越界、上下文过长和可能输出过多完整代码等风险。对于高风险请求，系统会将反馈降级为问题拆解、伪代码、局部检查、测试样例和反思提示，而不是直接输出完整程序。

系统评价维度包括解释准确性、修复选择性、认知适配性和反代写克制。该评价方式与论文中“正确、启发、克制、可用”的教育 AI 目标一致，强调智能辅导系统既要帮助学生解决问题，也要保留学生形成计算思维的过程。

## 8. 教学流程适配

系统支持 C-O-D-E-R 与 ADAPT 两类教学流程。C-O-D-E-R 流程对应情境激活、组织编排、协同开发、循环调试和反思重构，适合项目式编程课堂；ADAPT 流程对应诊断、支架、行动、练习和迁移，适合个性化伴学与课后辅导。通过流程参数切换，同一套图谱和智能体机制可以迁移到不同教学环节。

因此，CT-GraphMAS 不仅可以用于学生端即时问答，也可以服务于教师备课、课堂支架设计、课后错因分析、学习记录导出和知识库持续维护。

## 9. 原型实现

本原型采用 Python 标准库实现，不依赖外部 Web 框架或在线模型服务。后端使用 `http.server` 提供本地 Web 服务，使用 SQLite 保存文档、相关知识片段、节点、边和交互记录。前端采用 HTML、CSS 和 JavaScript 构建像素风界面，包含课堂参数设置、语料上传、三层对话、图谱画布、相关知识列表和门控日志。

系统当前使用确定性 hashing embedding 作为轻量检索实现，能够满足本地演示和结题原型需要。后续可替换为本地中文向量模型或在线 embedding API，以提升语义检索质量；也可将 SQLite 图谱迁移到 Neo4j，进一步支持复杂图推理和教师协同维护。
